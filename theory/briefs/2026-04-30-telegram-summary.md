2. 좋은 부분
- 프로그램/프로세스/스레드 기본 경계를 명확히 설명함
- TLB, ASID/PCID, CR3/TTBR로 원리 확장 질문을 수행함
- CPU/p99/error 조합으로 장애 가설을 세우는 관점이 있음

3. 부족한 부분
- race condition과 deadlock의 초기 혼동
- 조건부 대응(스레드 증설)을 일반화하는 경향
- 지표 기반 진단 순서의 고정 프레임 부족

4. 오늘 대화 주제
- 컨텍스트 스위칭, 스케줄링, 락 경합을 하나의 진단 체계로 묶기
- 기준: 증상 기반 가설 우선순위화, 지표로 검증, 반례로 기각 가능해야 함
- 실무 연결: 온콜에서 CPU 오진을 줄이고 MTTR을 낮추는 직접 역량

5. 오늘 질문
Q1. CPU 85~95%, p99 상승, 에러율 낮음 상황의 10분 진단 순서를 말해라.
Q2. CPU 20%, p99만 상승(평균 정상)일 때 원인 우선순위와 좁히는 절차를 설명해라.
Q3. race condition과 deadlock을 증상-원인-재현-완화 4축으로 비교해라.
Q4. 장애 격리 우선 시스템에서 프로세스 vs 스레드 선택 기준을 설명해라.
Q5. 컨텍스트 스위칭 비용을 과장 없이 설명해라(ASID/PCID 포함).
