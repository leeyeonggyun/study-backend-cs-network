# 오늘의 복습

- 어제는 클라이언트(요청을 먼저 시작하는 쪽)와 서버(요청을 받아 처리하는 쪽)를 역할로 구분했다.
- 요청과 응답을 한 번의 사이클로 보고, 문제를 단계별로 나눠 확인하는 관점을 배웠다.
- 프로그램(저장된 코드)과 프로세스(실행 중 상태) 구분이 요청-응답 문제 분석의 출발점이라는 점을 연결했다.
- 실무에서는 “응답이 이상하다”를 바로 코드 탓으로 단정하지 않고, 요청 시작/전달/처리/응답 반환 순서로 좁혀야 한다.

- 꼭 기억할 핵심:
  - “누가 먼저 요청을 시작했는가”가 클라이언트/서버 구분 기준이다.
  - “출력이 없다”와 “프로세스가 없다”는 완전히 다른 장애 유형이다.
  - 요청에는 목적, 응답에는 결과+이유가 있어야 점검이 가능하다.

- 약점/혼동 포인트 2개:
  - 앱이 멈춘 것처럼 보일 때 확인 순서(프로세스 존재 확인 → 입출력 흐름 확인)가 자주 뒤섞일 수 있다.
  - 출력을 터미널 글자만으로 좁게 이해해서, 파일/네트워크 응답/로그를 출력으로 놓치기 쉽다.

# 오늘의 신규 학습 주제

- 메인 주제: 인터넷, 도메인, IP, DNS 기초
- 이 주제를 고른 이유: 절대초보 리셋 경로에서 클라이언트/서버 다음 필수 단계다. 요청이 “어디로” 가는지 모르면 이후 HTTP 학습이 공중에 뜬다.
- 이전 학습과의 연결: 어제는 “누가 요청을 시작하는가”를 배웠고, 오늘은 “그 요청이 어떤 주소 체계로 목적지를 찾는가”를 배운다.
- 오늘 배워야 할 핵심 개념:
  - 인터넷: 여러 컴퓨터가 서로 통신하는 연결망
  - IP 주소: 실제 컴퓨터를 찾기 위한 숫자 주소
  - 도메인: 사람이 읽기 쉬운 이름 주소
  - DNS: 도메인을 IP로 바꿔주는 전화번호부 시스템
- 실무에서 왜 중요한지: “서버 코드 문제”인지 “이름 해석(DNS) 문제”인지 분리해야 장애 원인을 빠르게 좁힐 수 있다.

- 서브 주제: 포트의 의미와 기본 관찰
- 이 주제를 고른 이유: IP가 “컴퓨터 건물 주소”라면 포트는 “건물 안 서비스 출입구”라서, 다음 HTTP 학습 전에 반드시 감각이 필요하다.
- 이전 학습과의 연결: 어제 배운 클라이언트/서버 역할 위에, 오늘은 “같은 서버 컴퓨터 안에서 어떤 서비스로 들어갈지”를 포트로 연결한다.
- 오늘 배워야 할 핵심 개념:
  - 포트: 한 컴퓨터 안에서 서비스 프로세스를 구분하는 번호
  - listen: 서버 프로세스가 특정 포트에서 요청을 받을 준비 상태
  - connect: 클라이언트가 해당 포트로 접속 시도
- 실무에서 왜 중요한지: “서버는 떠 있는데 접속 안 됨” 상황에서 프로세스/포트/주소를 분리해 점검할 수 있다.

# 오늘의 학습 설명

메인 주제 설명:

1) 개념
- 인터넷은 “컴퓨터끼리 메시지를 주고받는 길”이다.
- IP(아이피)는 그 길에서 목적지를 찾기 위한 숫자 주소다. 예: 142.250.x.x
- 도메인은 사람이 외우기 쉬운 이름이다. 예: google.com
- DNS(도메인 네임 시스템)는 도메인을 IP로 바꿔주는 변환 시스템이다.
- 실제 흐름은 보통 “도메인 입력 → DNS 조회 → IP 획득 → 그 IP로 연결 시도” 순서다.

2) 흔한 오해
- 오해 1: “도메인 = 서버 자체”  
  -> 도메인은 이름이고, 실제 접속은 IP로 간다.
- 오해 2: “DNS가 느리면 서버 코드가 느린 것”  
  -> DNS 지연은 서버 애플리케이션 처리 전 단계 문제일 수 있다.
- 오해 3: “IP 하나면 서비스 하나”  
  -> 같은 IP에서 여러 서비스가 포트로 나뉜다(서브 주제 연결).

3) 실무 사용 예시
- 배포 직후 “내 API만 접속 안 됨”: DNS 레코드 갱신 여부 확인.
- 외부 API 호출 실패: 도메인 해석 실패인지(lookup 실패), 연결 실패인지 분리.
- 모니터링 경보: 애플리케이션 에러율과 DNS 실패율을 분리 관찰.

4) 트레이드오프/주의점
- 도메인은 사람이 편하지만, DNS 캐시 때문에 변경 반영이 즉시 안 될 수 있다.
- IP 직접 호출은 단기 점검엔 유용하지만, 운영에서는 도메인 기반 관리가 일반적이다.
- “핑이 된다”가 “서비스가 정상”을 보장하지 않는다(서비스 포트는 별개).

서브 주제 설명(간단):

- 포트는 한 컴퓨터(IP) 안에서 서비스 프로세스를 구분하는 번호다.
- 즉, IP가 같아도 포트가 다르면 다른 서비스다.
- 실무 예시: 같은 서버에서 22(SSH), 80(웹), 5432(DB)가 동시에 동작할 수 있다. 접속 실패 시 “IP 문제인지, 포트 미오픈인지”를 따로 본다.

# OpenHands 실습 브리프

- 실습 제목: DNS 조회와 포트 연결 관찰기 만들기
- 목표: 도메인→IP 변환과 포트 연결 성공/실패를 분리 관찰해, 네트워크 문제를 단계별로 진단하는 기초를 익힌다.
- 난이도: 입문 (Level 0)
- 사용 언어/스택: Python 3, 표준 라이브러리(socket, argparse), pytest
- 상세 요구사항:
  1) 도메인(예: `example.com`)을 입력받아 IP로 변환하는 CLI를 만든다.
  2) 변환된 IP에 대해 지정 포트(기본 80) 연결 가능 여부를 확인한다.
  3) 결과를 아래 3단계 상태로 출력한다:
     - `DNS_OK / DNS_FAIL`
     - `PORT_OPEN / PORT_CLOSED`
     - 최종 요약 메시지
  4) 에러 메시지는 초보자가 이해할 수 있게 한국어로 작성한다.
  5) 도메인 입력 시 `http://` 또는 `https://`가 붙어오면 호스트만 추출해서 처리한다.
- 예상 산출 파일:
  - `main.py`
  - `netcheck/dns_utils.py`
  - `netcheck/port_utils.py`
  - `tests/test_dns_utils.py`
  - `tests/test_port_utils.py`
  - `README.md`
- 의도적으로 넣을 버그/실패 케이스:
  - 초기 코드에 `http://example.com`을 그대로 DNS 조회하도록 남겨 테스트가 실패하게 만든다.  
    (학습자가 URL에서 호스트 추출 로직을 직접 고치게 함)
- 테스트 요구사항:
  1) 도메인 문자열 정규화 테스트(스킴 제거) 2개 이상
  2) DNS 성공/실패 분기 테스트 2개 이상(mock 사용 가능)
  3) 포트 연결 성공/실패 분기 테스트 2개 이상(mock 또는 로컬 임시 소켓)
  4) 총 6개 이상 테스트, `pytest -q` 기준 통과
- 이 실습으로 배우는 것:
  - 이름(도메인)과 실제 주소(IP)의 차이
  - DNS 단계 실패와 포트 단계 실패의 분리
  - “서버가 죽었나?”를 단정하지 않고 단계별 진단하는 습관
- 완료 체크리스트:
  - [ ] 도메인→IP 변환이 동작한다.
  - [ ] 포트 연결 결과가 열림/닫힘으로 구분된다.
  - [ ] URL 스킴 포함 입력(`https://...`)도 정상 처리된다.
  - [ ] 테스트 6개 이상이 모두 통과한다.
  - [ ] README에 실행 예시 2개 이상을 작성했다.
