# 오늘의 복습

- 어제는 인터넷 통신의 기본 단위(인터넷, 도메인, IP, DNS)를 배웠다.
- 도메인(이름 주소)과 IP(숫자 주소)의 역할이 다르고, DNS가 둘을 연결한다는 점을 확인했다.
- 포트는 같은 컴퓨터 안에서 어떤 서비스로 들어갈지 구분하는 번호라는 점을 배웠다.
- listen(서버의 대기)과 connect(클라이언트의 접속 시도)를 분리해서 봐야 장애 원인을 좁힐 수 있다.

- 꼭 기억할 개념:
  - DNS 문제와 애플리케이션 코드 문제는 분리해서 점검한다.
  - “프로세스가 떠 있음”과 “포트로 접속 가능함”은 같은 의미가 아니다.
  - 접속 실패 시 DNS -> 포트 -> 애플리케이션 순서로 좁혀간다.

- 약점/혼동 포인트 2개:
  - 도메인, IP, DNS를 개별 정의는 알아도 실제 장애 순서에 연결하면 섞이기 쉽다.
  - listen 상태와 실제 응답 가능 상태(앱 로직 정상)는 다르다는 점이 헷갈리기 쉽다.

# 오늘의 신규 학습 주제

- 메인 주제: HTTP 요청과 응답 기초
- 이 주제를 고른 이유: 어제까지 클라이언트/서버, DNS, 포트를 배워 HTTP를 이해할 최소 바닥이 준비됐다.
- 이전 학습과의 연결: DNS가 목적지 IP를 찾고, 포트로 연결된 뒤, 그 위에서 실제로 오가는 메시지가 HTTP 요청/응답이다.
- 오늘 배워야 할 핵심 개념:
  - HTTP(웹에서 요청/응답 형식을 정한 약속)
  - 요청(request)과 응답(response)의 한 사이클
  - HTTP 실패와 연결 실패를 구분하는 기준
- 실무에서 왜 중요한지: “서버가 안 된다”를 막연히 말하지 않고, 네트워크 문제인지 API 처리 문제인지 빠르게 분리할 수 있다.

- 서브 주제: 요청-응답 한 사이클 점검 루틴(복습 강화)
- 이 주제를 고른 이유: 초반에는 개념보다 점검 순서를 몸에 익히는 게 실무 효율에 더 직접적이다.
- 이전 학습과의 연결: 클라이언트/서버 역할 구분 + DNS/포트 점검을 실제 요청 한 건에 적용한다.
- 오늘 배워야 할 핵심 개념:
  - 한 요청을 “보냄 -> 도착 -> 처리 -> 응답”으로 나눠 추적
  - 각 단계에서 확인할 최소 신호(요청 로그, 응답 코드, 에러 메시지)
- 실무에서 왜 중요한지: 장애 대응 시 추측 대신 증거 기반으로 원인을 줄일 수 있다.

# 오늘의 학습 설명

메인 주제 설명: HTTP 요청과 응답 기초

- 개념 설명:
  - HTTP는 “웹에서 데이터를 주고받는 형식 규칙”이다.
  - 요청(request)은 클라이언트가 서버에 보내는 메시지다.
  - 응답(response)은 서버가 처리 결과를 돌려주는 메시지다.
  - 어제 배운 DNS/포트는 “길 찾기/문 찾기”이고, HTTP는 “문 안에서 무슨 말이 오갔는지”다.

- 흔한 오해:
  - 오해 1: HTTP 에러(예: 404)면 네트워크가 끊긴 것이다.
    - 실제: HTTP 응답을 받았다는 건 최소한 연결은 되었을 가능성이 높다.
  - 오해 2: 서버 프로세스가 떠 있으면 HTTP도 정상이다.
    - 실제: 프로세스가 살아도 라우트/핸들러 문제로 실패할 수 있다.
  - 오해 3: 요청 실패 원인은 항상 코드다.
    - 실제: DNS/포트/연결 단계 문제일 수도 있다.

- 실무 사용 예:
  - API 장애 티켓이 들어오면 먼저 “HTTP 응답 자체가 왔는지”를 본다.
  - 응답이 있으면 애플리케이션(라우팅/처리) 쪽, 응답 자체가 없으면 DNS/포트/연결 쪽을 우선 확인한다.
  - 로그에서 요청 ID/경로/응답 시간을 보면 병목 지점을 빠르게 찾을 수 있다.

- 트레이드오프/주의점:
  - 초기에 너무 많은 HTTP 세부 용어(method, header, body, status code)를 한 번에 외우면 오히려 혼란이 커진다.
  - 오늘은 “요청/응답 사이클을 분리해서 보는 습관”에 집중하고, 세부 요소는 다음 단계로 넘긴다.

서브 주제 설명: 요청-응답 한 사이클 점검 루틴

- 간단 개념:
  - 한 번의 호출을 4단계로 본다: 요청 보냄 -> 서버 도착 -> 서버 처리 -> 응답 반환.
  - 각 단계가 성공했는지 최소 증거를 하나씩 확보한다.

- 실무 예시 1개:
  - “사용자 목록 API가 안 열린다”면
    1) 클라이언트가 요청을 보냈는지(curl/브라우저 네트워크),
    2) 서버 로그에 요청이 찍혔는지,
    3) 응답 코드/본문이 무엇인지 확인해
    4) DNS/포트 문제인지, 서버 처리 문제인지 분류한다.
