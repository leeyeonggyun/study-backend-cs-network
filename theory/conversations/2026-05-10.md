## 1) Readiness timeout 이란?

주로 Kubernetes/로드밸런서 헬스체크에서 쓰는 말이야.

- **readiness probe**: “이 인스턴스가 지금 트래픽 받아도 되나?”
- **readiness timeout**: 정해진 시간 안에 readiness 체크 응답을 못 받음

즉, 프로세스가 떠 있어도(Alive)  
DB 연결/초기화/스레드 고갈 등으로 **서비스 준비가 안 된 상태**일 수 있고, 이때 timeout이 난다.

핵심:
- liveness 실패 = 죽었거나 hang → 재시작 대상
- readiness 실패 = 살아있지만 준비 안 됨 → 트래픽 제외 대상

---

## 2) Flapping 이란?

상태가 짧은 주기로 **정상↔비정상** 반복되는 현상.

예:
- readiness `Ready/NotReady`가 계속 왔다 갔다
- LB에서 backend up/down 반복
- 네트워크 링크 up/down 반복

문제점:
- 트래픽 라우팅이 흔들림
- 재시도 폭증, 지연 증가
- 장애가 “간헐적”으로 보여 디버깅 어려움

자주 있는 원인:
- timeout/threshold 설정이 너무 타이트
- GC pause, CPU 스파이크, cold start
- 종속 서비스(DB/Redis) 순간 지연
- 오토스케일링/재배포 중 워밍업 미흡

---

## 3) `connection refused` vs `timeout` vs `reset`  
### (A) TCP 단계 기준

### 3-1) Connection refused
- **TCP 3-way handshake 초입에서 실패**
- 클라이언트 SYN 보냄 → 대상에서 **RST 즉시 응답** (또는 OS가 거절)
- 의미: “그 포트에 받는 놈 없다/거부한다”

전형적 상황:
- 프로세스 미기동
- 해당 포트 미리스닝
- 잘못된 IP/포트
- 로컬 방화벽/보안정책이 reject 방식

---

### 3-2) Connection timeout
- 보통 **SYN 보냈는데 SYN-ACK를 못 받음** (침묵)
- 재전송하다 시간 초과
- 의미: “상대까지 못 갔거나, 중간/상대가 드랍”

전형적 상황:
- 네트워크 경로 단절
- Security Group/NACL/Firewall이 drop
- 라우팅 문제
- 대상 과부하로 응답 불가(커널 큐/네트워크 병목)

---

### 3-3) Connection reset (by peer)
- 연결 성립 후 또는 성립 도중에 **RST 패킷으로 강제 종료**
- 의미: “상대(또는 중간 장비)가 연결을 비정상 종료”

전형적 상황:
- 서버 프로세스 크래시/재시작 중
- Keepalive 불일치, idle timeout 후 재사용
- 프로토콜 mismatch (TLS 아닌데 TLS로 붙음 등)
- L7 프록시/LB가 정책상 RST

---

## 4) 원인 레이어로 분리해서 보기

### L3/L4 (네트워크/전송)
- 라우팅, 방화벽, SG/NACL, 포트 리스닝, SYN backlog
- 여기 문제면:
  - refused: 포트/리스너 부재 쪽 가능성 큼
  - timeout: 경로/드랍 가능성 큼
  - reset: 커널/중간장비가 끊음

### L7 (애플리케이션/프로토콜)
- TLS 핸드셰이크, HTTP keepalive, 프록시 타임아웃, 앱 크래시
- reset/readiness flapping은 L7 영향이 매우 흔함

### 인프라/오케스트레이션
- Kubernetes readiness/liveness 설정
- 롤링 업데이트, 오토스케일, 워밍업
- flapping의 구조적 원인 자주 여기서 나옴

- **Reset**: “문 열었다가 상대가 갑자기 쾅 닫음(RST)”

## 1) Threshold가 뭐냐
헬스체크에서 1번 실패했다고 바로 죽었다고 보지 않고,  
**N번 연속 실패/성공했을 때 상태를 바꾸는 기준값**을 둬.

대표적으로:
- `failureThreshold=3`  
  → 3번 연속 실패해야 `Unhealthy/NotReady`로 판정
- `successThreshold=2`  
  → 2번 연속 성공해야 `Ready`로 복귀

즉 threshold는 **노이즈(일시 지연, 순간 spike)**를 거르기 위한 완충장치야.

---

## 2) Flapping에서 “어떤 상태”가 왔다갔다하나
맞아, flapping은 상태가 짧게 반복 변하는 거고, 그 상태는 보통 헬스 상태야.

예시:
- `Ready ↔ NotReady` (K8s readiness)
- `Healthy ↔ Unhealthy` (LB health check)
- `UP ↔ DOWN` (모니터링 대상/링크 상태)

즉 “서버가 완전 죽고 사는 것”만 의미하는 게 아니라,  
**트래픽 받을 수 있냐 없냐의 판정 상태**가 계속 뒤집히는 걸 말해.

---

한 줄로:
- **threshold** = 상태 전환에 필요한 연속 성공/실패 횟수
- **flapping** = 그 상태 전환이 너무 자주 반복되는 현상

원하면 내가 `periodSeconds`, `timeoutSeconds`, `failureThreshold` 조합을 어떻게 잡아야 flapping 줄어드는지 실무값 예시로 바로 줄게.

## 1) 정상 연결 수립: 3-way handshake

클라이언트와 서버가 “연결 시작해도 되는지” 합의하는 과정.

```text
Client                    Server
  | ---- SYN(seq=x) ---->   |
  | <--- SYN,ACK(seq=y,ack=x+1) -- |
  | ---- ACK(ack=y+1) --->   |
  |      (ESTABLISHED)       |
```

의미:
1. **SYN**: “연결하고 싶어, 내 시작 번호는 x”
2. **SYN+ACK**: “좋아, 너 x 받았고(ack=x+1), 내 시작 번호는 y”
3. **ACK**: “너 y도 받았어(ack=y+1)” → 연결 성립

---

## 2) 정상 종료: FIN 기반 4-way 종료

양방향이 독립적으로 닫히기 때문에 보통 4단계.

```text
Client                         Server
  | ---- FIN(seq=u) --------->   |  (Client: 더 보낼 거 없음)
  | <--- ACK(ack=u+1) -------   |
  | <--- FIN(seq=v) ---------   |  (Server도 송신 종료)
  | ---- ACK(ack=v+1) ------->   |
  |         CLOSED/TIME_WAIT     |
```

포인트:
- **FIN**은 “정상적/질서 있는 종료”
- 상대가 남은 데이터 정리할 시간 있음

---

## 3) 비정상 종료: RST (강제 종료)

```text
Client                    Server
  | ---- data/SYN ----->    |
  | <--- RST ----------     |
```

상황:
- 포트 리스너 없음 (연결 시도 즉시 거절)
- 이미 끊긴 연결에 데이터 보냄
- 앱/커널/중간장비가 강제 종료

결과:
- `connection refused` (초기 연결 단계 거절)
- `connection reset by peer` (연결 중 강제 종료)

---

## 4) PSH/URG/ECE/CWR는 언제 보나?

### PSH
- “버퍼 더 모으지 말고 앱에 빨리 전달해줘”
- 대화형 트래픽(짧은 요청/응답)에서 흔히 보일 수 있음

### URG
- 긴급 포인터 유효 표시
- 현대 일반 웹서비스에서는 거의 안 씀

### ECE / CWR (ECN 혼잡제어)
- **ECE**: “혼잡 표시 받았어”
- **CWR**: “알겠고 혼잡윈도우 줄였어”
- 패킷 드랍 전에 혼잡 신호로 속도 조절하려는 메커니즘

---

## 5) 에러를 플래그 관점으로 빠르게 매핑

- **Connection refused** → 보통 `SYN`에 대한 `RST`
- **Connection timeout** → `SYN` 보냈는데 응답 없음(플래그 자체가 안 돌아옴)
- **Connection reset** → 연결 중 `RST` 수신
