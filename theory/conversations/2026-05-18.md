## 먼저 네가 말한 것 보정

- **Connection Refused**
  - 보통 `SYN` 보냈는데 대상에서 **즉시 RST**가 와서 거절된 케이스.
  - 즉, “요청은 갔고 상대가 거절했다”는 표현은 맞는데, TCP 레벨에서는 **거절 신호가 RST**로 오는 형태야.
  - 흔한 원인: 포트 미오픈, 프로세스 미리스닝, 중간 장비에서 reject.

- **Reset (RST)**
  - 이미 연결 중이든, 연결 시도 중이든, 상대/중간장비가 **강제 종료**.
  - “보내자마자 거절”도 reset이고, “데이터 주고받다가 갑자기 끊김”도 reset일 수 있음.

- **Timeout**
  - 요청(패킷)은 보냈는데 **기대한 응답이 시간 내 없음**.
  - 원인: 드랍(drop), 경로 문제, 방화벽 silent drop, 서버 행 걸림 등.

- **FIN**
  - TCP의 **정상 종료 절차**.
  - “대화 끝났으니 닫자”를 순서대로 주고받는 graceful close.

---

## FIN vs RST를 실무적으로 구분하는 법

### 1) FIN
- 의미: 정상적인 연결 종료
- 로그 느낌:
  - `FIN, ACK` → `ACK` → 반대쪽 `FIN, ACK` → `ACK`
- 해석:
  - 애플리케이션이 정상적으로 소켓 close
  - 보통 에러 아님

### 2) RST
- 의미: 비정상/강제 종료
- 로그 느낌:
  - `... RST` 또는 `connection reset by peer`
- 해석:
  - 포트 안 열림(초기 거절),
  - 앱 크래시/강제 종료,
  - LB/방화벽 정책 차단,
  - 프로토콜 불일치(예: TLS 포트에 평문 HTTP)

## “요청 문제 vs 응답 문제” 판별 프레임

너 질문의 핵심에 딱 맞게:

1. **SYN 자체가 나갔는가?**
   - 안 나가면 클라이언트/로컬 문제.
2. **SYN-ACK 또는 RST가 돌아왔는가?**
   - RST 즉시: 대상(또는 중간장비) 거절 = refused/reset 계열.
   - 아무것도 안 옴: timeout 계열(드랍/경로 문제).
3. **연결 성립 후 끊겼는가?**
   - FIN: 정상 종료.
   - RST: 비정상 강제 종료.
4. **HTTP 레벨 응답코드가 있는가?**
   - 4xx/5xx 있으면 TCP는 살아있고 **애플리케이션 응답** 문제.
   - 응답코드도 없고 소켓 에러면 **네트워크/TCP 레벨** 문제.

---

## “나머지” 자주 나오는 상태/에러

- **SYN timeout**
  - handshake 시작했는데 응답 없음.
- **Read timeout**
  - 연결은 됐는데 서버 데이터 응답 지연/없음.
- **Write timeout / Broken pipe**
  - 상대가 이미 닫았는데 내가 쓰려 함.
- **TLS handshake failure**
  - TCP는 됐지만 TLS 협상 실패(인증서, 버전, SNI 문제).
- **ICMP unreachable**
  - 네트워크/호스트/포트 도달 불가를 ICMP로 통지.
- **DNS failure**
  - 아예 목적지 IP 해석 실패 (`NXDOMAIN`, `SERVFAIL`).

---

## 한 줄 요약

- **FIN = 정상 종료**
- **RST/Refused = 거절 또는 강제 종료**
- **Timeout = 응답 미도착(대개 drop/지연/경로 문제)**

## Q1) `refused / reset / timeout / FIN` 정확히 구분

### 1) Connection Refused
- 보통 클라이언트가 `SYN` 보냄 → 서버(또는 중간장비)가 **즉시 RST**로 거절
- 의미: “그 포트로 받는 프로세스 없음 / 정책상 reject”
- 대표 원인
  - 프로세스 미기동
  - wrong port
  - 보안장비 reject rule
- 키워드: **즉시 거절, 빠른 실패**

### 2) Connection Reset (ECONNRESET)
- 이미 연결 시도/연결 중인 상태에서 상대가 **RST로 강제 종료**
- 의미: “정상 종료(FIN) 없이 끊김”
- 대표 원인
  - 앱 크래시/강제 close
  - LB/방화벽 idle timeout
  - 프로토콜 불일치(예: TLS 포트에 평문)
- 키워드: **강제 끊김**

### 3) Timeout
- 요청은 보냈는데 시간 내 응답 없음
- 의미: 패킷 드랍, 경로 블랙홀, 과부하로 응답 지연
- 대표 원인
  - 방화벽 drop
  - 네트워크 경로 문제
  - 서버 스레드풀 고갈
- 키워드: **무응답**

### 4) FIN
- TCP 정상 종료 신호
- 의미: 통신이 정상적으로 끝나서 graceful close
- 키워드: **정상 종료**

✅ 한 줄로:
- `refused` = 시작부터 즉시 거절  
- `reset` = 중간/직후 강제 종료  
- `timeout` = 응답 없음  
- `FIN` = 정상 종료

## Q2) 특정 AZ에서 `SYN-SENT` 급증 시 가설 3개 + 우선순위 + 지표

`SYN-SENT` 급증 = “연결 시작은 했는데 handshake 완료가 안 됨”

### 가설 A (우선순위 1): 네트워크/보안 경로 이슈 (drop/blackhole)
- 왜 1순위? SYN-SENT 증가는 handshake 미완료와 직결
- 볼 지표:
  - retransmission 증가
  - connect timeout 증가
  - NACL/SG 변경 이력
  - AZ 간/대외 경로 차이

### 가설 B (우선순위 2): 대상 포트 리스닝/헬스 이상
- 대상 인스턴스가 해당 포트 안 받거나 flap
- 볼 지표:
  - LB target health (AZ별 unhealthy 비율)
  - 인스턴스별 listening port 상태
  - 재기동/배포 시점 일치 여부

### 가설 C (우선순위 3): 포화/고갈 (conntrack, backlog, ephemeral port)
- 연결 요청을 받아도 처리 못함
- 볼 지표:
  - SYN backlog overflow
  - conntrack near limit
  - 서버 CPU run queue / accept queue
  - NAT GW/ephemeral port 사용량

실무 순서:
1) **네트워크 drop 여부**  
2) **리스너/헬스 상태**  
3) **고갈/포화**

## Q3) readiness timeout 과도 단축 → flapping → 5xx 증가 인과

### 인과 체인
1. readiness timeout을 너무 짧게 설정  
2. 잠깐 느려진 파드가 “NotReady” 판정  
3. LB/Service에서 target 제거  
4. 곧 회복되면 다시 Ready로 붙음  
5. 이게 반복(flapping)되며 target pool 출렁임  
6. 순간적으로 healthy target 부족 → 5xx(502/503/504) 증가

### 왜 위험?
- 실제 장애가 아닌 “탐지 민감도 과대”로 장애를 유발
- autoscaling/rollout 중 더 악화됨

### 완화
- readiness timeout/period/failureThreshold 재조정
- startupProbe 분리 (초기 warm-up 보호)
- preStop + terminationGracePeriod으로 drain 보장
- LB deregistration delay/slow start 조정

## Q4) 장애 발생 후 10분 진단 체크리스트 (멀티 AZ timeout 급증)

### 0~2분: 현상 분리
- AZ별 에러율/지연 분리
- `connect timeout` vs `read timeout` vs `5xx` 분리
- 클라이언트 공통인지 특정 서비스만인지 확인

### 2~5분: 연결 계층 확인
- SYN-SENT/retransmission/Reset 비율 확인
- LB target health를 AZ별로 확인
- 최근 SG/NACL/Route/배포 변경점 확인

### 5~8분: 빠른 완화
- 문제 AZ target weight 0 또는 일시 제외
- retry 정책 점검(무한 재시도 금지, jitter 적용)
- 타임아웃 재조정(너무 짧은 connect/read 완화)

### 8~10분: 근본 원인 후보 고정
- 네트워크 drop인지, 리스너 문제인지, 포화인지 1차 분류
- 의존성(DB, cache, NAT, DNS) 단일 실패점 여부 확인
- RCA용 증적(지표/로그/패킷) 저장

---

## Q5) Keepalive의 트레이드오프 + 실패 신호

### 장점
- handshake 비용 감소
- 지연 개선
- 포트/CPU 절약(짧은 연결 폭증 대비)

### 단점/리스크
- 죽은 연결(stale connection) 재사용 위험
- LB/NAT idle timeout과 불일치 시 reset 증가
- 일부 backend 불균형(롱커넥션 편향)

### 실패 신호
- `connection reset by peer` 급증
- 첫 요청만 실패(재시도 시 성공)
- 특정 주기마다 에러 spike (idle timeout 주기와 일치)
- FIN보다 RST 비율이 비정상 증가

### 운영 팁
- client keepalive timeout < LB/NAT idle timeout 로 정렬
- connection pool max idle / max lifetime 설정
- 재시도는 idempotent 요청에만 제한적으로
