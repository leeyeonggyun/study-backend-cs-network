## 피드백 (짧고 정확하게)

네 답변 요지:
- 프로세스 죽음 → 다른 프로세스 영향 적음 (격리)
- 스레드 죽음 → 프로그램 전체 영향 큼 (자원 공유)

### 정확한 표현으로 보정
1) **프로세스**
- 맞음: 주소공간/FD 테이블 등 기본 자원이 프로세스 경계로 격리됨.
- 단, “영향이 아예 없음”은 아님.  
  예: 죽은 프로세스가 DB 커넥션 풀, 락, IPC, 큐 소비자 역할이었다면 **간접 영향** 발생.

2) **스레드**
- “스레드 하나 죽으면 무조건 전체 종료”는 **케이스 바이 케이스**.
- 같은 프로세스 내 메모리를 공유하므로 한 스레드의 버그(메모리 오염, fatal signal)는 전체를 망가뜨리기 쉬움.
- 하지만 단순히 worker thread 하나가 정상적으로 종료되는 건 프로세스 전체 종료와 동일하지 않음.

  - 스레드들이 락 대기(block) 상태라 runnable이 줄고 CPU는 한가함
  - 요청은 큐에 쌓여 p99 증가

- **원인2: race condition으로 인한 재시도/롤백/오류 경로 진입**
  - 직접 CPU를 많이 쓰기보다, 실패 후 대기·재시도로 tail latency 악화
