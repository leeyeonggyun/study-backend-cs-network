## 1) 개념 보정 (중요 포인트만)

### ✅ 프로그램 / 프로세스 / 스레드
- **프로그램**: 디스크에 있는 실행 파일(정적 상태)
- **프로세스**: 실행 중인 프로그램 인스턴스(메모리 올라온 상태, 자원 보유)
- **스레드**: 프로세스 내 실행 흐름 단위(실제 CPU 스케줄링 대상)

여기까지 아주 정확합니다.

---

### ✅ 프로세스 vs 스레드 자원
- **프로세스 간**: 기본적으로 메모리 공간 분리(격리)
- **스레드 간**: 같은 프로세스의 코드/데이터/힙 공유, 스택/레지스터는 각자 가짐

맞습니다.

### ⚠️ “같은 바이너리 두 번 실행하면 공유 없음”
이건 *거의 맞지만 표현 보완*이 필요해요.

- 논리적으로는 **서로 다른 프로세스**라서 주소공간 분리됨
- 하지만 OS 최적화로
  - 실행 코드(text segment), 공유 라이브러리 페이지는 **물리 메모리 레벨에서 공유 가능**
  - 파일 캐시도 공유 가능

즉, **프로세스 관점에선 분리**, **커널/물리 메모리 관점에선 일부 공유 최적화 가능**.

## 2) 동시성 이슈 부분 (가장 중요한 수정)

작성하신 문장:
> 같은 자원에 동시에 접근하면 레이스 경쟁이 생겨서 무한대기됨

이건 두 개가 섞였어요.

- **Race Condition(경쟁 상태)**  
  동기화 없이 공유 데이터 접근 → 결과가 실행 순서에 따라 달라짐(데이터 깨짐/비일관성)
- **Deadlock(교착 상태, 무한대기)**  
  락 획득 순환 대기 등으로 서로 영원히 기다림

즉,  
**Race condition ≠ Deadlock**  
레이스가 난다고 항상 무한대기 되는 건 아닙니다.

## 3) 성능 관측 해석 (CPU 90%, 지연↑, 에러율 낮음)

가설 방향 좋습니다. 실무적으로는 이렇게 말하면 더 좋아요:

- 증상: **CPU 사용률 높음 + latency 증가 + error 낮음**
- 1차 가설:
  1) CPU-bound 병목
  2) lock 경합/스핀 증가
  3) GC/런타임 오버헤드
- 대응:
  - 스레드 수를 무조건 늘리기 전에
    - CPU run queue, context switch, lock wait, GC time 먼저 확인
  - CPU-bound면 스레드 과증가는 오히려 악화 가능
  - I/O-bound면 스레드/워커 증가가 효과적일 수 있음

즉 “스레드 늘린다”는 **조건부 전략**입니다.

브라우저 탭별 프로세스 예시는 매우 좋은 예시입니다.

## 1) 프로그램 / 프로세스 / 스레드 — 면접 합격형 정의

## 1-1. Program (프로그램)
- 디스크에 저장된 **정적 바이너리 + 메타데이터**
- 아직 실행 컨텍스트(레지스터, 스택, 열린 FD, PID)가 없음

## 1-2. Process (프로세스)
- 실행 중인 프로그램의 **보호된 실행 컨테이너**
- 핵심: **독립 가상주소공간 + 커널이 관리하는 자원 집합**
  - 예: PID, page table, file descriptors, signal handlers, credentials, cwd 등
- 프로세스는 “실행 단위”라기보다 **격리/보호 경계(isolation boundary)** 에 더 가깝다

## 1-3. Thread (스레드)
- 프로세스 내부의 **실제 실행 흐름(schedulable entity)**
- 각 스레드는 자체:
  - PC(프로그램 카운터), 레지스터, 스택, TLS
- 같은 프로세스 스레드끼리는 공유:
  - code/text, heap, global data, 열린 파일 FD

## 2) “같은 바이너리 두 번 실행”의 정확한 해석

네 말처럼 **서로 다른 프로세스**가 생성되어 주소공간은 분리됨(정답).  
하지만 상위 답변으로 가려면 이걸 추가:

- 각 프로세스는 **독립 virtual address space** 보유
- 다만 물리 메모리 레벨에서는 커널이 최적화:
  - read-only code page/shared library page는 **공유 매핑 가능**
  - fork 후엔 COW(copy-on-write)로 실제 쓰기 시점에만 분리

즉:
- **보안/격리 관점:** 분리
- **물리 메모리 효율 관점:** 일부 공유

이걸 말하면 면접관이 “아, VM/메모리 모델 이해하네”라고 봅니다.

## 3) 컨텍스트 스위칭 비용 — 왜 스레드가 대체로 유리한가

`thread switch` vs `process switch`를 질적으로 설명해야 합니다.

- 스레드 전환(같은 프로세스 내):
  - 주로 레지스터/스택 포인터/스케줄링 상태 저장·복원
  - 같은 주소공간이라 TLB flush 부담이 상대적으로 작을 수 있음
- 프로세스 전환:
  - 주소공간 전환(page table context change)
  - TLB 영향(아키텍처/최적화에 따라 다름), 캐시 locality 저하 가능

중요:  
“스레드가 무조건 빠름”이 아니라 **워크로드/락경합/캐시미스**까지 포함해 봐야 함.

## 4) 동시성 버그 구분 (매우 자주 나오는 면접 함정)

네 문장 정정:

- **Race condition**
  - 공유 데이터 접근 순서가 비결정적이라 결과가 깨짐
  - 예: `x++` 동시 실행 시 lost update
- **Deadlock**
  - 락 획득 순환대기 등으로 영원히 블록
- **Livelock**
  - 서로 양보만 하며 진행 없음
- **Starvation**
  - 특정 스레드가 계속 CPU/락을 못 받음

면접에서 “race가 무한대기 만든다”라고 하면 감점 가능.  
정확히 분리해서 말해야 합니다.

## 5) CPU 90%, latency↑, error 낮음 — SRE/백엔드 해석 프레임

이 해석 프레임으로 답하면 수준 높아 보입니다:

1. **증상**
   - 높은 CPU + 높은 p95/p99 latency + 낮은 error
2. **가설**
   - CPU-bound 계산 병목
   - lock contention/spin
   - GC pause/allocator pressure
   - 과도한 context switching
3. **검증 지표**
   - run queue length
   - context switches/sec
   - lock wait time / mutex contention
   - CPU profile (hot function)
   - GC time %, stop-the-world pause
4. **조치**
   - CPU-bound: 알고리즘/벡터화/샤딩/수평확장 우선
   - I/O-bound: 워커/스레드 증가 효과
   - 락경합: lock granularity 축소, lock-free 구조 검토
   - 스레드 수는 **실험 기반 튜닝**, 무작정 증가는 금물

## 6) 프로세스 모델 vs 스레드 모델 선택 기준 (설계 면접용)

## 프로세스 분리가 유리할 때
- 장애 격리 최우선 (한 컴포넌트 크래시가 전체 전파되면 안 됨)
- 보안 경계 필요 (권한/샌드박스 분리)
- 런타임/GC/의존성 분리 필요

예: 브라우저 탭, DB worker 분리, 플러그인 샌드박스

## 스레드가 유리할 때
- 저지연 공유 메모리 접근 필요
- 데이터 교환이 매우 잦아 IPC 비용이 부담
- 동일 신뢰 경계 내 병렬 처리

예: 웹서버 worker pool, in-process parallel compute

## 7) 면접용 “메모리 구조 그림 기반” 설명

아래 ASCII로 설명하면 화이트보드에서 강합니다.

```text
[Disk]
  └─ myapp (program: static binary)

실행 #1 -> Process A (PID 1001)
실행 #2 -> Process B (PID 1002)

Process A virtual memory
+---------------------------+
| Code/Text (R-X)           |  <-- 일부는 Process B와 물리페이지 공유 가능
| Global/Data (RW-)         |
| Heap (shared by threads)  |
| Stack T1                  |  <-- Thread 1 private
| Stack T2                  |  <-- Thread 2 private
+---------------------------+
FD table, signal handlers, page table, credentials...

Process B virtual memory
+---------------------------+
| Code/Text (R-X)           |  <-- 논리적 주소공간은 분리
| Global/Data (RW-)         |
| Heap                      |
| Stack T1                  |
+---------------------------+
```

스레드 관점:

```text
Process A
  ├─ Thread 1: PC/Registers/Stack1 (private)
  ├─ Thread 2: PC/Registers/Stack2 (private)
  └─ Shared: Code, Heap, Global, FD
```

핵심 문장:
- “스레드는 실행 컨텍스트를 분리하고, 프로세스는 보호 경계를 분리한다.”

## 8) 60초 면접 답변 버전 (암기용)

“프로그램은 디스크의 정적 실행파일이고, 프로세스는 그 프로그램이 실행되며 커널이 관리하는 독립 자원 단위입니다. 프로세스마다 가상주소공간이 분리되어 장애와 보안 격리에 유리합니다. 반면 스레드는 프로세스 내부 실행 흐름으로, 스택과 레지스터는 분리되지만 힙과 전역 메모리를 공유해 통신 비용이 낮고 컨텍스트 전환이 비교적 가볍습니다. 그래서 고성능 병렬 처리에는 스레드가 유리할 수 있지만 race condition, deadlock 같은 동기화 위험이 커집니다. 설계 시에는 ‘격리/안정성’이 우선이면 멀티프로세스, ‘저지연 공유와 처리량’이 우선이면 멀티스레드를 선택하고, 실제로는 브라우저처럼 프로세스 격리 위에 탭 내부 스레드를 쓰는 하이브리드 구조를 많이 사용합니다.”

## 2) 프로세스와 스레드의 본질적 차이 (정의보다 중요한 것)

면접에서 고평가 받는 문장:

> **프로세스는 ‘보호 경계’이고, 스레드는 ‘실행 경계’다.**

- 프로세스: 주소공간/권한/자원 격리
- 스레드: 같은 주소공간 안에서 실행 흐름을 늘려 병렬성 확보

### 메모리 관점
- 프로세스 간: 기본 분리
- 스레드 간: 힙/전역 공유, 스택 분리

### 실패 격리 관점
- 프로세스 분리: 한쪽 죽어도 다른 쪽 살 가능성 높음
- 스레드: 한 스레드의 메모리 오염이 전체 프로세스 리스크

## 3) 주소변환 심화: TLB, 페이지테이블, 왜 비싼가

CPU는 가상주소 → 물리주소 변환이 항상 필요.

- 페이지테이블 다단계 워크는 느림
- 그래서 **TLB(주소변환 캐시)** 사용

### 컨텍스트 스위칭 때 비용 포인트
1. 주소공간 전환 (CR3/TTBR 변경)
2. TLB 유효성 문제(태그 불일치 엔트리 배제/flush 필요)
3. 캐시(locality) 재학습 필요 (L1/L2/L3, branch predictor 등)

맞는 포인트:
- “프로세스 스위칭 시 캐시/변환 캐시 측면에서 손해”는 정확함

정밀 포인트:
- ASID/PCID로 완화 가능 (완전 flush 빈도 감소)

## 4) 프로세스 스위칭이 스레드 스위칭보다 보통 비싼 이유

단순히 “무겁다”가 아니라 이유를 말해야 함:

- 주소공간 전환 비용
- TLB hit rate 저하
- 캐시 워밍업 손실
- 스케줄링/커널 bookkeeping 증가 가능

단, 예외:
- 같은 프로세스 내 스레드라도 락경합/false sharing 심하면 매우 느려질 수 있음  
  (즉 “스레드=무조건 빠름”은 오답)

## 5) 동시성 함정 4종 세트 (면접 단골)

1. **Race condition**
   - 순서 비결정성으로 데이터 훼손
2. **Deadlock**
   - 순환대기, 영구 블록
3. **Livelock**
   - 바쁘게 움직이지만 진전 없음
4. **Starvation**
   - 특정 작업이 계속 기회 못 얻음

## 6) 성능 이슈를 볼 때의 실무 프레임 (네가 물은 CPU 90% 케이스)

해석: “장애”보다 “포화/병목” 가능성이 큼

확인 순서:

2. Run queue 길이
3. lock contention
4. GC pause/time
5. context switch rate
6. syscall 비율 / kernel time

조치 원칙:
- CPU-bound: 스레드만 늘리면 악화 가능
- I/O-bound: 워커/스레드 증가 유효 가능
- 락경합: 자료구조/락 설계 재검토가 우선

## 7) 빅테크 면접에서 좋아하는 “한 단계 더” 포인트

## 7-1. 멀티프로세스인데도 공유하는 것
- 물리 페이지 공유(text/shared lib, page cache)
- COW fork 최적화

## 7-2. IPC 비용과 선택
- shared memory (빠름, 동기화 복잡)
- socket/pipe (단순, 복사/커널 경유 비용)

## 7-3. NUMA
- 메모리가 CPU 소켓별로 멀고 가까움
- 스레드-메모리 배치가 틀어지면 latency 폭증

## 7-4. False sharing
- 서로 다른 변수라도 같은 cache line이면 캐시 invalidation ping-pong
- 멀티스레드 성능 망가짐

## 1) 가상메모리 심화 (Virtual Memory Deep Dive)

## 핵심 개념
- 각 프로세스는 **독립 가상주소공간(VA)** 을 가짐
- MMU가 VA→PA(물리주소) 변환
- 변환 캐시는 **TLB**
- 실제 페이지는 필요할 때 올리는 **Demand Paging**
- `fork()` 후 즉시 복사 안 하고 **COW (Copy-On-Write)**

## 면접 포인트
1. **왜 가상메모리?**
   - 격리/보안, 메모리 효율, 큰 주소공간 추상화
2. **페이지 폴트 종류**
   - Minor fault: 페이지는 메모리에 있으나 매핑만 없음
   - Major fault: 디스크 I/O 필요(느림)
3. **COW 장점**
   - fork 직후 빠름, 메모리 절약
4. **mmap**
   - 파일 I/O와 메모리 접근 통합, zero-copy 패턴 가능

## 실무에서 보는 지표
- page fault rate (major/minor 분리)
- RSS vs VSZ
- swap in/out
- OOM kill 기록
- TLB miss 관련 성능 카운터(가능한 환경에서)

## 흔한 문제
- 메모리 릭 -> RSS 꾸준히 증가
- swap thrash -> 지연 급증
- huge page 미적용/오남용 -> 성능 출렁임

---

## 2) 스케줄러 심화 (Scheduler Deep Dive)

## 핵심 개념
- 스레드는 run queue에서 CPU 배정 대기
- 리눅스 CFS는 “공정성” 중심(가상 런타임)
- **선점(preemption)** 과 **우선순위**가 지연에 큰 영향

## 면접 포인트
1. **CPU-bound vs I/O-bound**
   - CPU-bound는 코어 수/스케줄링 영향 큼
   - I/O-bound는 대기시간 최적화가 핵심
2. **Load average / run queue**
   - 큐 길면 대기 지연 증가
3. **Priority inversion**
   - 낮은 우선순위가 락 잡아 높은 우선순위가 대기
4. **CPU affinity / NUMA**
   - 잘못 배치되면 캐시/메모리 원격접근으로 tail latency 상승

## 실무 지표
- run queue length
- context switches/sec
- CPU utilization (user/system/iowait)
- throttling(cgroup quota hit)
- per-core imbalance

## 흔한 문제
- 스레드 과다 생성 -> 스위칭 오버헤드 증가
- 컨테이너 CPU quota 낮음 -> p99 악화

---

## 3) 락킹 심화 (Locking & Concurrency)

## 핵심 개념
- 임계구역 보호: mutex/spinlock/rwlock/semaphore
- 동기화 없으면 race condition
- 락 설계 실패하면 deadlock/livelock/starvation

## 면접 포인트
1. **mutex vs spinlock**
   - mutex: 잠들고 깸(대기 길면 유리)
   - spin: 바쁜 대기(아주 짧은 락에서 유리)
2. **RWLock**
   - read 많고 write 적을 때 효과적
3. **Lock granularity**
   - coarse lock(쉬움, 병목 가능)
   - fine lock(복잡, 성능 잠재력)
4. **False sharing**
   - 다른 변수라도 같은 cache line이면 성능 저하
5. **메모리 모델**
   - atomic, memory ordering(acquire/release) 이해 중요

## 실무 지표
- lock wait time
- blocked thread 수
- contention hot lock
- critical section 길이
- retry/spin 횟수

## 흔한 문제
- 락 순서 불일치 -> deadlock
- 거대한 전역 락 -> throughput 한계
- lock-free 남용 -> 디버깅 악몽

---

## 4) Perf 실전 (성능분석 실무 루틴)

## 문제 유형 분류
1. CPU↑ + p99↑ : 연산/스케줄링 포화
2. CPU↓ + p99↑ : 대기/의존성 tail
3. CPU↓ + p99정상 + error↑ : 기능/배포/의존성 실패

## 실전 절차 (온콜형)
1. **증상 고정**
   - 언제부터, 어떤 endpoint, 어떤 AZ/노드
2. **골든 시그널**
   - latency(p50/p95/p99), error, traffic, saturation
3. **프로파일링**
   - CPU flamegraph/hot path
4. **대기 분석**
   - DB/API latency, queue wait, lock wait, GC pause
5. **노드 편차 확인**
   - 특정 인스턴스만 느린지
6. **실험**
   - 스레드/커넥션풀/쿼리/캐시 정책 변경 A/B
7. **회귀 방지**
   - SLO 알람/대시보드/부하테스트 케이스 추가

## 자주 쓰는 액션
- p99 상승 시: timeout + retry 정책 먼저 점검(무한 retry 금지)
- lock contention 시: shard/partition, 자료구조 분리
- CPU hot path: 알고리즘 개선 > 스레드 증설

## 3) 오늘 보정된 포인트 (중요 교정)

### ⚠️ 1) Race condition vs Deadlock

- 보정:

  → 서로 다른 문제

- 조건부로만 맞음

Tutor: 좋은 비유인데, **정확히는 이렇게 보는 게 맞아요**:

- **Hermes agent(나)**: 하나의 “에이전트 실행 컨텍스트”
- **subagent**: “내부 스레드”라기보다 **분리된 워커(거의 별도 프로세스에 가까운 모델)**

왜냐면 subagent는:
1) 대화 컨텍스트가 분리되고  
2) 터미널 세션도 분리되고  
3) 중간 작업 상태를 직접 공유하지 않기 때문이에요.

즉 OS 비유로 하면  
- `main agent ↔ subagent`는 **멀티스레드**보다 **멀티프로세스 + IPC**에 더 가깝습니다.

한 줄 결론:  
**“subagent를 스레드라고 보기보단, 격리된 별도 프로세스형 워커로 보는 게 더 정확”**합니다.
