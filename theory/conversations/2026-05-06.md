## 1) SYN-SENT / ESTABLISHED가 뭐야?

TCP 연결 상태(state) 이름이야.

- **SYN-SENT**
  - 클라이언트가 서버에 `SYN`(연결 시작) 보냄
  - 아직 서버의 `SYN-ACK`를 못 받았거나, 받아도 최종 `ACK`까지 안 된 상태
  - 즉, **“연결 시도 중”**

- **ESTABLISHED**
  - 3-way handshake 완료
  - 실제 데이터 송수신 가능한 상태
  - 즉, **“연결 성공”**

그래서:
- `SYN-SENT↑`, `ESTABLISHED↔`면  
  → 연결 성공률이 떨어지거나, 연결 성립이 지연되는 쪽 의심.

---

## 2) 3-way handshake가 뭐야?

TCP 연결 시작 절차:

1. Client → Server: `SYN`  
2. Server → Client: `SYN-ACK`  
3. Client → Server: `ACK`  

이 3단계가 끝나야 `ESTABLISHED`.

중간에 패킷 손실/드랍/큐 포화가 있으면 연결이 늦거나 실패함.

---

## 3) Retransmission(재전송)이 뭐야?

TCP는 신뢰성 보장 프로토콜이라,
- 보낸 패킷 ACK를 못 받으면 **같은 패킷을 다시 보냄** = 재전송

재전송률이 높다는 건 보통:
- 네트워크 품질 저하
- 중간 장비/큐 포화로 드랍
- 서버가 제때 응답 못함  
중 하나.

실무에선 latency/timeout 증가와 같이 나타남.

---

## 4) Keepalive가 뭐야?

“한 번 맺은 연결을 재사용할지”와 관련된 설정.

- **HTTP keep-alive / upstream keepalive (Nginx)**
  - Nginx가 백엔드(User-Service)와 매 요청마다 새 TCP 연결을 만들지 않고
  - 기존 연결을 풀에 보관했다가 재사용
  - 장점: handshake 비용 감소, latency 감소, 커넥션 폭주 완화

설정이 잘못되면:
- 연결 재사용이 안 되어 매번 새 연결 생성
- 그러면 SYN 폭증, 지연 증가 가능

---

## 5) LB / NAT가 뭐야?

- **LB (Load Balancer)**
  - 요청을 여러 서버로 분산
  - L4(LB가 TCP 수준에서 분산) / L7(HTTP 기준 라우팅) 등이 있음

- **NAT (Network Address Translation)**
  - 내부 IP를 외부 IP로 바꾸거나 그 반대
  - 클라우드/쿠버네티스/노드 경계에서 흔히 사용됨

문제 포인트:
- NAT 테이블/포트 고갈, LB 설정 불일치가 있으면
- 신규 연결이 지연/실패할 수 있음 (`SYN-SENT` 증가로 보일 수 있음)

---

## 6) readinessProbe timeout이 뭐야?

Kubernetes에서 Pod가 **“트래픽 받아도 되는 준비 상태인지”** 검사하는 헬스체크.

- `readinessProbe` 실패하면 그 Pod는 Service endpoint에서 제외됨
- `timeoutSeconds`를 너무 짧게 잡으면,
  - 순간 느려진 정상 Pod도 “준비 안 됨”으로 판단
  - 실제 healthy pod가 자주 빠졌다 들어왔다(flapping)
  - 결과: 일부 요청이 연결 실패/timeout을 경험

즉, 어제 변경한 readiness timeout 단축은 이번 장애와 **강한 연관 후보**야.

---

## 7) upstream은 뭐야?

Nginx 기준으로:
- 클라이언트 입장: Nginx가 서버
- Nginx 입장: 뒤에 붙은 실제 앱 서버(User-Service)가 **upstream**

그래서 로그의
- `connecting to upstream`
- `upstream timed out`
는 “Nginx ↔ User-Service 구간” 문제를 뜻함.

## 가설 1 (우선순위 최고):  
**Nginx ↔ User-Service 신규 TCP 연결이 비정상적으로 많아졌고(keepalive 이슈), 연결 성립 단계에서 병목이 발생했다.**

### 1) 왜 유력한가
- `SYN-SENT 급증 + ESTABLISHED 정체`는 전형적으로 **연결 성립 전 구간 문제** 시그널.
- 어제 `Nginx upstream keepalive 설정 변경`이 있었음(직접적인 변화점).
- keepalive가 깨지면 요청마다 새 TCP handshake 필요 → SYN 폭증 → 지연/timeout/일부 refused로 연결될 수 있음.
- CPU/메모리/GC는 정상이라 “애플리케이션 연산 병목”보다 “연결/네트워크 경로 병목” 가능성이 큼.

### 2) 어떻게 검증할까
- **Nginx 상태/메트릭**
  - upstream keepalive hit/miss, active/idle upstream connection 수
  - `$upstream_connect_time`, `$upstream_header_time`, `$request_time` 분포 비교
- **Nginx 설정 diff 확인**
  - `upstream { keepalive N; }`
  - `proxy_http_version 1.1;`
  - `proxy_set_header Connection "";` (잘못 `close` 되지 않았는지)
- **소켓 상태 확인**
  - Nginx 노드에서 `ss -s`, `ss -tan | grep SYN-SENT | wc -l`
  - User-Service Pod/노드에서도 SYN backlog 관련 지표 확인
- **에러로그 상관관계**
  - `connect() failed`, `upstream timed out` 시점과 keepalive 변경 배포 시점 비교

## 가설 2:  
**readinessProbe timeout 단축으로 endpoint flapping이 발생해, Nginx가 불안정/준비안된 Pod로 연결 시도하고 실패했다.**

- 변경사항에 `readinessProbe timeout` 단축이 명시됨.
- readiness가 너무 빡세면 순간적인 지연에도 Pod가 Ready↔NotReady 반복(flapping).
- 그 순간 Service endpoint 집합이 흔들리고, Nginx는 방금 제외/재등록되는 Pod로 연결 시도하다
  - `connection refused` (프로세스가 아직 listen 준비 전)
  - `upstream timeout` (느린 초기화/응답 지연)
  을 유발할 수 있음.
- replica를 8→16으로 늘린 직후라, 새 Pod warm-up 문제와 결합되면 더 심해짐.

### 2) 어떻게 검증할까
- **K8s 이벤트/상태**
  - `kubectl describe pod <pod>`에서 readiness probe failed 이벤트 빈도
  - `kubectl get endpoints/endpointslice` 변화 빈도(짧은 주기 흔들림)
- **프로브 설정 검토**
  - `timeoutSeconds`, `periodSeconds`, `failureThreshold`, `initialDelaySeconds`
  - startupProbe 유무 (초기 구동 느린 앱 보호)
- **애플리케이션 로그**
  - readiness endpoint 응답시간/p99, 타임아웃 로그
  - Pod 시작 직후 DB connection pool warm-up 지연 여부
- **상관분석**
  - NotReady 전환 타임스탬프와 5xx spike, Nginx connect error 증가 시점 일치 여부

## 가설 3:  
**LB/NAT/노드 커널 레벨에서 연결 자원(포트/conntrack/backlog) 병목 또는 패킷 드랍이 발생했다.**

### 1) 왜 유력한가
- 재전송률 상승은 네트워크 중간 구간 품질/드랍/큐 포화를 강하게 시사.
- `SYN-SENT` 증가 + timeout은 L4 경로에서 SYN/SYN-ACK 왕복 손실에도 잘 나타남.
- 스케일아웃(8→16) 후 연결 패턴이 바뀌면 conntrack/NAT 테이블 압박이 생길 수 있음.
- 앱 리소스는 정상인데 연결 실패가 늘면 인프라 네트워크/커널도 반드시 의심해야 함.

### 2) 어떻게 검증할까
- **노드 커널/네트워크 지표**
  - `netstat -s` 또는 `ss -s`에서 retransmit, listen queue overflow
  - `/proc/net/netstat`의 `ListenOverflows`, `ListenDrops` 계열
- **conntrack/NAT 상태**
  - conntrack 사용량 vs 최대치 (`conntrack -S`, sysctl max)
  - NAT 포트 고갈 징후(특정 source NAT IP 집중 여부)
- **패킷 캡처(짧게)**
  - SYN은 나가는데 SYN-ACK가 안 오는지, 오는 ACK가 손실되는지
- **LB 메트릭**
  - backend connection error, reset, health-check flap, zone imbalance
- **노드별 편차**
  - 특정 노드/AZ에서만 실패율 높으면 경로/네트워크 이슈 가능성 증가

## 1) endpoint flapping

**의미**  
Kubernetes에서 어떤 Pod가 Service의 endpoint에 **붙었다 빠졌다를 짧은 주기로 반복**하는 현상.

- Ready 되면 endpoint에 포함
- NotReady 되면 endpoint에서 제외
- 이게 반복 = flapping

**왜 위험한가**  
트래픽 라우팅 대상이 계속 바뀌어서:
- 연결 실패
- timeout
- 순간적인 5xx 증가

주원인: readinessProbe가 너무 엄격(특히 timeout 너무 짧음), 앱 warm-up 지연, 의존성(DB/외부 API) 순간 장애.

---

## 2) replica

**의미**  
동일한 Pod(서비스 인스턴스) 개수.

예: replica=8 → 같은 앱 Pod 8개 동작.  
8→16으로 늘리면 처리량 여유는 늘 수 있지만, 동시에:
- 새 Pod 초기화 부하
- endpoint 변동 증가
- 연결 패턴 변화
가 생길 수 있음.

---

## 3) retransmit (TCP retransmission)

**의미**  
TCP가 보낸 패킷의 ACK를 못 받았을 때 **같은 패킷 재전송**하는 것.

**많아지면 뜻하는 바**
- 패킷 손실/드랍
- 네트워크 혼잡
- 중간 장비(LB/NAT/방화벽) 문제
- 수신측 과부하로 응답 지연

즉, “연결은 있는데 품질이 나쁘다” 혹은 “연결 성립 중 손실 있다”의 신호.

---

## 4) listen queue overflow

먼저 배경:
서버 프로세스가 `listen()` 하면 커널이 “들어오는 연결”을 큐로 잠깐 쌓아둠.

이 큐가 꽉 찼는데 새 연결이 오면 = **listen queue overflow**.

결과:
- 신규 연결 지연/실패
- client 쪽에서 timeout, 재시도, SYN-SENT 증가 가능

보통 순간 트래픽 스파이크, accept 처리 지연, backlog 사이즈 부족에서 발생.

---

## 5) ListenOverflows

리눅스 TCP 통계 카운터 중 하나.  
**listen 큐가 꽉 차서 overflow 난 횟수**를 의미.

값이 증가하면:
- 서버가 새 연결을 제때 못 받고 있다는 강력한 증거.

---

## 6) ListenDrops

역시 리눅스 TCP 통계 카운터.  
listen 단계에서 들어온 연결이 **드랍된 횟수**.

ListenOverflows와 함께 증가하면 보통:
- backlog 포화
- accept 지연
- 연결 처리 병목
을 의심.

---

## 한 번에 연결해서 이해하기

장애 때 이런 흐름이 자주 나와:

`readiness flapping` 또는 `keepalive 미스`  
→ 신규 TCP 연결 급증  
→ listen queue 압박  
→ `ListenOverflows/ListenDrops` 증가  
→ handshake 지연/실패 (`SYN-SENT` 증가, retransmit 증가)  
→ 최종적으로 timeout/5xx 증가

Tutor: 좋아. Kubernetes에서 **Pod**은 제일 중요한 기본 단위야.

## 한 줄 정의
**Pod = 쿠버네티스에서 배포되는 가장 작은 실행 단위**  
(보통 컨테이너 1개, 가끔 여러 개를 묶음)

---

## 왜 Pod가 필요해?
쿠버네티스는 컨테이너를 직접 다루기보다 **Pod 단위**로 스케줄링/복구/네트워킹을 해.

즉:
- “컨테이너 하나 실행”이 아니라
- “Pod 하나 실행(그 안에 컨테이너들 포함)”이라고 생각하면 맞아.

---

## Pod의 핵심 특징 4개

1. **같은 Pod 안 컨테이너는 IP를 공유**
   - Pod마다 고유 IP 1개
   - 같은 Pod 내부 컨테이너끼리는 `localhost`로 통신 가능

2. **같은 네트워크/스토리지를 공유 가능**
   - volume 공유해서 파일 같이 사용 가능

3. **생명주기를 같이 가짐**
   - Pod가 죽으면 안의 컨테이너도 같이 재시작/재생성

4. **일시적(ephemeral)**
   - Pod는 교체될 수 있는 존재
   - “특정 Pod에 영구히 의존”하면 안 됨

---

## 컨테이너 vs Pod
- **컨테이너**: 앱 실행 프로세스
- **Pod**: 컨테이너를 감싸는 실행 껍질(네트워크/스토리지/정책 포함)

실무에서는 보통:
- Pod 하나에 앱 컨테이너 1개
- 필요하면 sidecar(로그 수집, 프록시 등) 추가

---

## 장애 문맥에서 왜 중요해?
네가 본 `readinessProbe`, `replica`, `endpoint`가 다 Pod 중심 개념이야.

- readinessProbe 실패 → 그 **Pod**는 트래픽 대상에서 빠짐
- replica=16 → 같은 역할 Pod가 16개
- endpoint flapping → Pod들이 Ready/NotReady 반복하며 라우팅 대상이 흔들림
