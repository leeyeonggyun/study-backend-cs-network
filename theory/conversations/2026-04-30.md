## 0) MTTR이 뭐야?

**MTTR (Mean Time To Recovery/Repair)**  
- 장애가 발생한 시점부터 **정상 상태로 복구될 때까지 걸린 평균 시간**.
- 온콜 관점에서는 보통 아래를 포함:
  1) 탐지 (Detect)  
  2) 원인 파악 (Diagnose)  
  3) 완화/복구 (Mitigate/Recover)  
  4) 사후 재발방지 (Follow-up)

실무에서 MTTR을 줄인다는 건:
- “CPU 높다” 같은 피상 증상에서 헤매는 시간을 줄이고
- **가설 우선순위**를 빨리 정해서
- **검증/반증**을 빨리 수행해
- 가장 빠른 완화조치를 먼저 하는 것.

---

## 1) 왜 CPU 오진이 자주 나오는가

온콜에서 “CPU 90%”만 보고 CPU 자체를 원인으로 착각하기 쉬움.  
하지만 실제로는:

- **컨텍스트 스위칭 폭증** → CPU는 바쁜데 “유효 작업”은 적음  
- **스케줄링 지연(run queue 증가)** → 요청 지연/타임아웃  
- **락 경합(lock contention)** → 스레드들이 lock 기다리며 낭비

즉, CPU 사용률은 **결과(증상)**인 경우가 많고,  
원인은 “스케줄러/락/동시성 설계” 쪽인 경우가 많음.

---

## 2) 세 가지를 하나로 묶는 진단 프레임: **CSR-L 프레임워크**

여기서 CSR-L =  
- **C**ontext Switch
- **S**cheduling
- **R**un Queue (스케줄링 압력)
- **L**ock Contention

### 핵심 아이디어
CPU 이슈를 “단일 지표”로 보지 말고,  
**4개 축을 동시에 관측하고 가설 기반으로 우선순위화**한다.

---

## 3) 온콜용 진단 순서 (5~10분 버전)

## Step 1) 증상 분류 (30~60초)
- 사용자 영향: latency/p99, error rate, timeout 여부
- 시스템 영향: CPU%, load avg, run queue, context switch rate

→ 여기서 목표: “CPU 자체 문제냐, 동시성/스케줄링 문제냐”의 1차 분기.

## Step 2) 초기 가설 3개 세우기
예시:
- **H1**: 락 경합이 root cause다.
- **H2**: 스케줄링 압력(과한 runnable thread/process)이다.
- **H3**: 진짜 계산량 증가(정상적인 CPU-bound workload 증가)다.

## Step 3) 우선순위 점수화 (증거 기반)
가설마다 점수:
- **Impact** (영향 크기) 1~5
- **Likelihood** (가능성) 1~5
- **Time-to-Verify** (검증시간, 짧을수록 고점) 1~5

`Priority Score = Impact × Likelihood × VerifySpeed`

온콜은 “정확도 100%”보다 **빠른 반증 가능성**이 중요.

## Step 4) 빠른 검증(반증 포함)
각 가설마다 “지지 지표 + 반례 지표”를 같이 본다.

## Step 5) 완화조치 먼저, 근본원인 추적은 그다음
MTTR 단축의 핵심:
- 서비스 살리는 조치(스레드풀/동시성 제한/롤백/트래픽 분산)를 먼저
- RCA는 안정화 후 깊게

---

## 4) 가설별 “지표-검증-반례” 템플릿

---

### H1) 락 경합이 원인

**보는 지표**
- lock wait time, mutex contention, blocked thread 수
- CPU 높지만 user-time 대비 sys-time/idle 패턴 불균형
- 스레드 덤프에서 동일 락 대기 체인 반복

**검증**
- 상위 핫락(top contended lock) 식별
- 해당 락 소유 코드 경로와 임계구역 길이 확인
- 임시로 concurrency 줄여 contention 감소 확인

**반례 (H1 기각 신호)**
- 락 wait 거의 없음
- blocked thread 적고 runnable만 많음
- lock-free 구간에서도 latency 동일

**즉시 완화 예시**
- 요청 동시성 제한 (backpressure)
- hot shard 분산
- 락 범위 축소/임계구역 밖으로 I/O 이동(후속 조치)

---

### H2) 스케줄링/런큐 압력이 원인

**보는 지표**
- run queue length 급증
- context switch/sec 폭증 (특히 voluntary/involuntary 패턴 변화)
- CPU migration, scheduler latency
- thread 수가 코어 수 대비 과도

**검증**
- 최근 배포에서 thread pool/worker 설정 변경 여부
- 스레드 폭증 서비스/프로세스 식별
- 동시성 캡 적용 시 p99 개선되는지 A/B 확인

**반례**
- run queue 정상
- context switch 낮음
- 동시성 줄여도 latency 개선 거의 없음

**즉시 완화**
- worker/thread pool downsize
- 큐 길이 제한 + 드롭 정책
- noisy neighbor 격리(코어/노드 분리)

---

### H3) 실제 CPU-bound workload 증가

**보는 지표**
- user CPU time 증가가 주요 원인
- 프로파일링 시 특정 함수가 명확히 hot path
- context switch/lock 지표는 상대적으로 안정적

**검증**
- 최근 트래픽/요청 종류 변화 확인
- flamegraph/profiler로 상위 CPU 함수 확인
- 해당 연산(압축, 암호화, 직렬화, 정렬 등) 비중 증가 확인

**반례**
- hot function이 없고 wait 지표만 높음
- 요청량은 비슷한데 지연만 증가
- lock/scheduler 지표와 함께 악화

**즉시 완화**
- 고비용 기능 feature flag off
- 캐시 강화
- 샘플링/배치화/정밀도 완화

---

## 5) “CPU 오진 줄이는” 우선순위 규칙 (실무형)

온콜 룰을 이렇게 고정하면 오진이 줄어듦:

1. **CPU% 단독으로 결론 금지**  
2. 반드시 세트로 본다:  
   - `CPU% + run queue + context switches + lock wait`
3. 5분 내 **최소 1개 가설 기각**을 목표로 한다.
4. 완화조치는 “검증 가능한 것”부터:
   - 동시성 제한 → 효과 있으면 scheduler/lock 계열 가능성↑
5. 포스트모템에 “오진 지표” 기록:
   - 처음 세운 가설, 틀린 이유, 다음번 fast-check 항목

---

## 6) 온콜 플레이북 예시 (짧은 체크리스트)

## 0~2분
- 고객 영향 확인 (p95/p99, 에러율)
- CPU 외 3지표(runQ, cs/s, lock wait) 동시 확인

## 2~5분
- H1/H2/H3 점수화
- 가장 빨리 반증 가능한 가설부터 실행

## 5~10분
- 완화조치 1개 적용(동시성 제한 등)
- 지표 개선 여부로 가설 업데이트
- 필요 시 롤백/트래픽 분산

## 10분+
- 안정화되면 깊은 RCA
- 재발방지 액션: 모니터링/알람/코드 수정

---

## 7) 지표 설계 (장기적으로 MTTR 낮추기)

대시보드를 “원인축”으로 구성:

- **CPU 패널**
  - user/system/iowait/steal 분해
- **Scheduler 패널**
  - run queue length
  - context switch rate (voluntary/involuntary)
- **Lock 패널**
  - lock wait total
  - top contended locks
  - blocked threads
- **Service 패널**
  - p95/p99 latency, timeout, error rate

그리고 알람도 단일 CPU 알람 대신 복합조건:
- CPU high **AND** run queue high
- CPU high **AND** lock wait high

이렇게 해야 “CPU 높음 = 무조건 스케일아웃” 같은 오판을 줄임.

---

## 8) 한 줄 요약

- **MTTR** = 장애 복구 평균 시간  
- CPU 오진을 줄이려면 CPU를 결과지표로 보고,  
  **컨텍스트 스위칭 + 스케줄링 + 락 경합**을 하나의 가설 기반 진단체계(CSR-L)로 운영해야 함.  
- 핵심은 **가설 우선순위화(영향×가능성×검증속도)** + **반례 기반 기각** + **빠른 완화조치**.

Tutor: **온콜(On-call)**은  
서비스 장애나 긴급 이슈가 났을 때 **즉시 대응하도록 대기하는 근무 체계**야.

쉽게 말해:
- 평소엔 개발/운영 업무를 하다가
- 온콜 시간에는 알람(장애, 에러율 급증, 지연 증가 등)이 오면
- 담당자가 바로 확인하고 복구까지 책임지는 역할.

보통 포함되는 일:
1. 알람 확인  
2. 원인 진단  
3. 임시 완화/복구  
4. 필요하면 다른 팀 에스컬레이션  
5. 사후 보고(재발 방지 액션)

즉, **“문제 생기면 가장 먼저 받고 가장 먼저 움직이는 담당자”**라고 보면 돼.
