## 8. 저녁 회고

### 오늘 배운 핵심을 3문장으로 설명
1. HTTP 요청은 URL/Method/Header/Body로 분해해 확인하면 디버깅이 빨라진다.
2. 응답이 아예 없을 때와 4xx/5xx가 올 때는 문제 계층이 다르므로 진단 순서를 분리해야 한다.
3. 200/400/404를 빠르게 읽으면 장애 범위를 즉시 줄일 수 있다.

### 오늘 가장 어려웠던 점
- 400과 404의 경계를 실제 케이스로 즉시 판단하는 부분.

### 오늘 실습에서 가장 중요한 포인트
- 클라이언트 입력 오류를 400으로 일관되게 반환하도록 검증 규칙과 테스트를 고정하는 것.

### 실제 서비스/실무에 연결하면
- 요청 분해와 상태코드 해석이 API 장애 분류와 원인 추적 시간을 단축한다.

### 내일 아침 복습 큐
- POST /users가 400일 때 URL/Method/Header/Body 점검 3가지를 말한다.
- 404가 경로/자원 문제인지 확인하는 절차를 말한다.
