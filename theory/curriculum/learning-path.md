# Backend, CS, Network Learning Path

목표: 백엔드, CS, 네트워크를 전혀 모르는 상태에서 시작해, 용어를 이해하고, 작은 서버를 만들고, 로그와 테스트를 읽고, 점차 실무 개념으로 올라간다.

## 핵심 원칙

- 시작점은 절대 초보다. HTTP, TCP, 프로세스, 포트, API 같은 단어도 모를 수 있다고 가정한다.
- 하루에 새로운 개념 점프는 하나만 한다.
- 어제 배운 개념을 오늘 다시 써먹게 만든다.
- 아직 모르는 용어를 전제로 다음 단계로 넘어가지 않는다.
- 어려운 주제를 빨리 건드리는 것보다, 연결이 끊기지 않게 배우는 것을 우선한다.

## 진행 방식

1. 용어 이해: 오늘 처음 보는 단어를 짧게 정의한다.
2. 흐름 이해: 개념이 어디에서 쓰이는지 연결한다.
3. 작은 확인: 퀴즈나 간단한 실습으로 직접 확인한다.
4. 기록: 헷갈린 용어와 막힌 지점을 로그에 남긴다.
5. 반복: 다음날 같은 흐름 위에 한 단계만 올린다.

## 단계별 커리큘럼

### Phase 0: 컴퓨터와 프로그램 기초

- 컴퓨터가 프로그램을 실행한다는 뜻
- 프로그램과 프로세스
- 파일, 폴더, 실행
- 터미널 명령의 의미
- 입력과 출력

### Phase 1: 인터넷과 웹의 가장 바닥

- 클라이언트와 서버
- 요청과 응답
- 인터넷에 연결된다는 의미
- 도메인, IP, DNS
- 포트의 의미

### Phase 2: HTTP 기초

- 브라우저가 서버에 요청하는 흐름
- URL이 무엇인지
- method, path, query
- header와 body
- 기본 status code
- JSON이 왜 필요한지

### Phase 3: 백엔드 입문

- 서버 프로그램이 요청을 받는 과정
- 라우팅
- 요청 검증
- 에러 응답
- 로그 기초
- 테스트 기초

### Phase 4: 데이터와 저장

- 데이터베이스가 필요한 이유
- 테이블, row, column
- CRUD
- 간단한 SQL
- 트랜잭션 입문

### Phase 5: 네트워크와 운영 기초

- TCP 연결
- timeout
- keep-alive
- connection pool
- TLS/HTTPS 기초

### Phase 6: 실무 기초 패턴

- 인증/인가 기초
- session과 JWT
- Redis cache와 TTL
- retry와 idempotency
- rate limit

## 주제 연결 규칙

- 프로그램/프로세스를 모르면 서버 프로세스로 넘어가지 않는다.
- 클라이언트/서버를 모르면 HTTP로 넘어가지 않는다.
- 도메인/IP/DNS/포트를 모르면 네트워크 장애 분석으로 넘어가지 않는다.
- HTTP 요청/응답을 모르면 REST, 인증, 캐시, timeout으로 넘어가지 않는다.
- status code를 모르면 에러 계약, retry, idempotency로 넘어가지 않는다.
- 로그와 테스트를 모르면 장애 대응 주제로 넘어가지 않는다.
- CRUD를 모르면 transaction, index, slow query로 넘어가지 않는다.
